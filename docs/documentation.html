1. Обзор проекта

Маркетплейс — это полнофункциональная многосторонняя платформа электронной коммерции (multi-vendor marketplace) с уклоном в handmade / крафтовые товары. Помимо классической продажи физических товаров платформа поддерживает два дополнительных бизнес-направления: цифровые товары с мгновенной доставкой (файлы, лицензии) и встроенную LMS — продажу и прохождение онлайн-курсов с видео, уроками, тестами и сертификатами.

Проект реализует полный жизненный цикл торговли: витрину и каталог, корзину и оформление заказа, онлайн-оплату с фискализацией (54-ФЗ), доставку через российских логистических операторов, разделение платежей между продавцами, вывод средств, возвраты и споры, поддержку, маркетинговые инструменты (промокоды, акции, продвижение, программы лояльности и рефералов), а также богатую админ-панель и отдельные кабинеты для продавцов.

~38Kстрок кода (backend + frontend)
85таблиц БД
347API-эндпоинтов
50сервисов бизнес-логики
~96экранов фронтенда
5микросервисов + worker
Целевая аудитория и роли. В системе пять ролей пользователей: buyer — покупатель seller — продавец support — поддержка moderator — модератор superadmin — суперадмин. Права внутри админки дополнительно гранулируются через RBAC (набор разрешений на пункты меню).

3. Архитектура

Изначально проект был монолитом на FastAPI (~16K строк). Была проведена поэтапная (staged) миграция:

  • Этап 1: перевод с MySQL на PostgreSQL, адаптация под инфраструктуру Yandex Cloud.
  • Этап 2: распил на микросервисы поверх общей БД (shared-database microservices) — 5 независимо разворачиваемых FastAPI-сервисов, разделённых по доменам, плюс Celery worker/beat.
Ключевое архитектурное решение. Выбрана модель «сервисы на общей Managed PostgreSQL», а НЕ классические микросервисы «БД на сервис». Общий код вынесен в устанавливаемую библиотеку shared/app (пакет app) — модели, схемы, сервисы, конфигурация переиспользуются всеми сервисами без дублирования. Каждый сервис — свой Docker-образ и своя запись маршрута в Kong.

Фронтенд — отдельный статический контейнер (nginx), обращается к API по относительному /api/v1 (тот же origin через Kong), поэтому CORS не требуется. Kong отдаёт catch-all маршрут / на фронтенд, а более длинные префиксы (/api/v1/*, /health, /uploads) выигрывают и уходят на backend.

Навигация

Админка и кабинет продавца используют группированное боковое меню (sider) из сворачиваемых секций; в админке пункты фильтруются по RBAC-правам пользователя. Меню покупателя — сгруппированный выпадающий список у аватара.

11. Кабинет продавца на отдельном поддомене

Кабинет продавца вынесен на отдельный вход на поддомене seller.<домен> с собственным sider-меню в стиле админки (6 групп: Обзор / Товары / Продажи / Маркетинг / Финансы / Магазин).

Как устроено (Вариант 1 — единый бандл, host-aware)

  • Тот же фронтенд-контейнер и бандл обслуживают и витрину, и поддомен. На старте SPA проверяет window.location.hostname; если он начинается с seller. — рендерится отдельное приложение продавца (свой логин + SellerLayout) вместо витрины.
  • Внутри кабинета — корневые URL: seller.<домен>/products, /orders и т.д. Старые ссылки <домен>/seller/* редиректят на поддомен.
  • Отдельный вход отклоняет не-продавцов; авторизация в localStorage пер-origin, поэтому вход на поддомене независим от витрины.

Инфраструктура — только DNS

Kong catch-all / и /api/v1/* маршрутизируются по пути (host-agnostic), поэтому тот же контейнер и backend уже обслуживают поддомен. Управление DNS вынесено в Terraform (infra/terraform/dns.tf, включается переменной domain_name): создаётся зона Cloud DNS + A-записи апекса и seller. на один и тот же IP балансировщика. Роль dns.editor у СА Terraform.

На будущее (Вариант 2). Код изолирован (SellerApp — самостоятельное дерево роутов), поэтому переезд на отдельную сборку/контейнер с собственным маршрутом Kong по хосту будет правкой сборки, а не переписыванием.

12. Инфраструктура и деплой

Terraform

Вся инфраструктура описана в infra/terraform: сеть (public/private подсети), Managed PostgreSQL 17 и Redis/Valkey, Compute Instance Groups для Kong и backend, Network Load Balancer, target-группы, Object Storage (2 бакета + статический ключ), Postbox (SMTP), Cloud DNS, а также автогенерируемый Ansible-инвентарь hosts.ini со всеми секретами и адресами.

Ansible

Плейбуки в infra/ansible: deploy-all.yml (CA БД → сервисы+миграции+Kong → фронтенд одной командой), deploy-services.yml (5 сервисов + worker/beat + маршруты Kong), deploy-frontend.yml (независимый контейнер фронта + catch-all Kong), kong-setup.yml (bootstrap Kong на Postgres), observability.yml (Prometheus + Grafana на бастионе).

CI/CD

GitHub Actions: ci.yml (компиляция backend + pytest, сборка фронта) и deploy.yml (сборка и пуш 6 backend-образов + фронтенда в Yandex Container Registry, опциональный передеплой Ansible). Скрипт build-and-push.sh собирает и пушит 7 образов.

Env-контракт. Ansible инжектит в контейнеры строгий набор переменных: DATABASE_URL и DB_* (пулер :6432), REDIS_URL, SECRET_KEY/REFRESH_SECRET_KEY/JWT_ALGORITHM, S3_* и SMTP_*, FRONTEND_URL. Миграции запускаются одноразовым контейнером (migrate), образ backend диспетчеризует команды migrate | web | worker | beat | seed.

13. Безопасность

  • JWT-аутентификация с раздельными секретами для access- и refresh-токенов; авто-обновление токена на 401 через перехватчик axios.
  • RBAC в админке: набор разрешений определяет доступные пункты меню и эндпоинты (rbac_service, эндпоинт «моё меню»).
  • Двухфакторная аутентификация (2FA) — отдельный сервис и экраны.
  • KYC / верификация продавцов (verification_service, trust_service): статусы доверия, VIP, документы хранятся в приватном бакете.
  • Разделение хранилищ: публичные ассеты и приватные (цифровые товары, HLS, KYC) — разные бакеты; приватные отдаются только по presigned-URL, анонимный доступ = 403.
  • Rate limiting (core/ratelimit.py), хеширование паролей bcrypt.
  • Аудит действий (audit_log) для чувствительных операций админки.
  • TLS БД: поддержка DB_SSL + корневой сертификат Yandex для Managed PostgreSQL.
Открытый пункт. На публичном контуре пока нет HTTPS на самом edge: NLB работает на L4 (TCP), TLS предполагается терминировать на Kong отдельным шагом (сертификат wildcard *.<домен> или SAN с поддоменом). Grafana развёрнута с анонимным доступом — допустимо для демо, требует закрытия в проде.

14. Функциональные возможности

14.1 Покупатель

  • Каталог с фильтрами, атрибутами, вариантами товара, сравнением, вопросами к товару.
  • Корзина, оформление заказа, несколько адресов доставки, выбор оператора доставки.
  • Онлайн-оплата с чеком 54-ФЗ; рассрочка (BNPL).
  • Заказы и суб-заказы (по разным продавцам), отслеживание статусов, возвраты, споры.
  • Цифровые товары: мгновенная выдача файлов, личная библиотека покупок.
  • Онлайн-курсы: плеер с HLS-видео, уроки, тесты, прогресс, сертификаты с проверкой по коду.
  • Избранное, коллекции (вишлисты), подписка на магазины, подписки на товары («снова в наличии»/«цена упала»).
  • Программа лояльности (уровни/баллы), рефералы, подарочные сертификаты и промо-баланс.
  • Индивидуальные заказы: запрос на изготовление → расчёт продавцом → частичная предоплата.
  • Чат с продавцом, уведомления, тикеты поддержки, мультивалютность, RU/EN.

14.2 Продавец

  • Дашборд, управление товарами, импорт из CSV, склад и движения остатков, варианты/атрибуты.
  • Заказы и суб-заказы, возвраты, споры со стороны продавца.
  • Аналитика продаж, отзывы на товары и ответы на них.
  • Маркетинг: собственные промокоды, флеш-распродажи, акции и наборы (bundle), платное продвижение с рекламным кошельком.
  • Финансы: вывод средств, тариф и комиссия (подписки продавца), налоговые реквизиты и режим (СНО).
  • Сотрудники магазина (роли внутри магазина), KYC / доверие / VIP.
  • Академия продавца (обучающие курсы для продавцов), конструктор курсов, шаблоны чата и часы работы.

14.3 Модератор / Администратор

  • Управление пользователями и магазинами (360°-карточки), верификация продавцов.
  • Модерация товаров и очередь модерации, категории, отзывы, жалобы.
  • Заказы, выплаты, фискальные чеки, реконсиляция платежей.
  • Маркетинг платформы: купоны, баннеры, рассылки, сертификаты, уровни лояльности, рефералы.
  • Настройки платформы, тарифы, платные возможности, валюты, feature-flags, SMS.
  • Аналитика платформы, когорты и LTV, метрики Grafana, журнал аудита.

14.4 Поддержка

  • Стол поддержки (тикеты с SLA и авто-назначением), арбитраж споров между покупателем и продавцом.

15. Рекомендации: что есть и что можно улучшить

Что реализовано сильно

  • Широта функционала: физические + цифровые товары + LMS, полный цикл денег (оплата → фискализация → расщепление → выплаты → возвраты), богатый маркетинг.
  • Чистая поэтапная миграция монолита в сервисы на общей БД без переписывания импортов (общая библиотека app).
  • Инфраструктура как код целиком: Terraform + Ansible + CI/CD, воспроизводимый деплой в Yandex Cloud.
  • Продуманная деградация интеграций (S3/SMTP/поиск/SMS) — удобно для разработки.
  • Разделение публичных/приватных хранилищ и presigned-выдача цифровых товаров.

Приоритетные улучшения

Инфраструктура и эксплуатация

  • HTTPS/TLS на edge — сейчас публичный контур по HTTP. Терминировать TLS на Kong (wildcard-сертификат, включая поддомен продавца), включить HSTS.
  • Закрыть Grafana (сейчас анонимный доступ) и вынести мониторинг за аутентификацию.
  • Строгая сборка фронтенда: сейчас прод собирается через vite build в обход tsc из-за оставшихся ошибок типов — починить типы и вернуть tsc && vite build.
  • Зафиксировать зависимости фронтенда (нет package-lock.json → CI использует npm install); перейти на npm ci для воспроизводимости.
  • Секреты — вынести в управляемое хранилище (Yandex Lockbox / Vault) вместо инъекции через Ansible-inventory.

Архитектура и данные

  • Общая БД — узкое место при росте: ввести схемы-на-сервис и явные контракты, чтобы позже безболезненно разнести данные (движение к Варианту 2 и для фронта продавца).
  • Единая монорепа-библиотека app связывает сервисы на уровне кода — по мере роста выделять доменные пакеты с версионированием.
  • Добавить нагрузочное тестирование и явные индексы под тяжёлые аналитические запросы (когорты, реконсиляция).

Качество и надёжность

  • Расширить автотесты (сейчас минимальный pytest в CI) — контрактные тесты API и e2e ключевых сценариев (оплата, возврат, выдача цифрового товара).
  • Идемпотентность платёжных вебхуков и повторных попыток выплат; сверка с ОФД/эквайрингом.
  • Централизованное структурированное логирование + трассировка запросов между сервисами (trace-id через Kong).

Продукт и рост (отложенный roadmap)

Уже согласованы к постройке ближайшие пункты: рассрочка (BNPL), индивидуальные заказы, подарочная упаковка/сертификаты, Академия продавца, расширенные отзывы (видео, бейдж проверенной покупки, галерея). Отложено (рекомендуется вернуться после):

  • AI-слой: чат-бот поддержки, авто-генерация описаний/тегов, семантический/визуальный поиск, персональная лента.
  • Рекламный кабинет продавца с аналитикой ставок/охвата/ROI, A/B-тесты карточек, прогноз спроса.
  • Live-шопинг и короткие видео (reels).
  • Платная подписка покупателя (бесплатная доставка + привилегии) на базе лояльности.
  • Escrow-бейдж (выплата при завершении заказа ≈ escrow — сделать это явным преимуществом).
  • Fulfilment-light (склад маркетплейса, консолидированная доставка).
  • Мобильные приложения / PWA.
  • B2B-режим (прайс-листы, счета, опт) и трансграничная торговля (i18n-контент, таможня, мультивалютные расчёты).

UX и мелочи

  • Единый onboarding продавца (KYC → реквизиты → тариф) как пошаговый мастер.
  • Сквозной поиск и подсказки на витрине через Meilisearch (включить в проде).
  • Доступность (a11y) и мобильная адаптация ключевых экранов чекаута и кабинета продавца.
