# 🛍️ Marketplace — Wildberries × Etsy

Полнофункциональный маркетплейс с прямыми продавцами, реферальной программой,
гибкими комиссиями и полной админ-панелью. Backend на FastAPI (async),
frontend на React + TypeScript, очереди на Celery + Redis, БД — **PostgreSQL**
(Yandex Managed PostgreSQL в проде).

> **Архитектура: микросервисы на общей БД.** Backend разбит на shared-библиотеку
> (`shared/`) и 5 независимо деплоящихся сервисов (`services/identity`, `catalog`,
> `orders`, `sellers`, `platform`) + worker. Маршрутизация — через API-gateway
> (Kong в проде, nginx локально) по префиксам пути. См. [STAGE2_PLAN.md](STAGE2_PLAN.md)
> и [infra/STAGE2_INFRA_CHANGES.md](infra/STAGE2_INFRA_CHANGES.md). Развёртывание —
> Yandex Cloud (Terraform + Ansible), БД и Redis — managed-сервисы.

```
┌─────────────┐     ┌──────────────┐     ┌──────────────────────────┐
│   Frontend   │ ──▶ │  gateway     │ ──▶ │ identity / catalog /     │
│  React+Vite  │     │ (nginx/Kong) │     │ orders / sellers /       │
│  (nginx:3000)│ ◀── │   :8000      │ ◀── │ platform  (FastAPI)      │
└─────────────┘     └──────────────┘     └────────────┬─────────────┘
                                                       │
                              ┌────────────────────────┼───────────────┐
                              ▼                         ▼               ▼
                        ┌───────────┐            ┌──────────┐    ┌──────────┐
                        │ PostgreSQL│            │  Redis   │    │  Celery  │
                        │    17     │            │ (broker) │    │ worker/  │
                        │ (общая БД)│            └──────────┘    │  beat    │
                        └───────────┘                           └──────────┘
                              │
                ┌─────────────┴────────────┐
                ▼                          ▼
          ┌───────────┐            ┌──────────────────────┐
          │ YooKassa  │            │ CDEK / Ozon / Яндекс  │
          │  (оплата) │            │ / Почта (доставка)    │
          └───────────┘            └──────────────────────┘
```

**Почему так:**
- **FastAPI + SQLAlchemy 2.0 async** — высокая производительность под нагрузкой, неблокирующий I/O для платежных вебхуков и запросов к внешним API.
- **Celery + Redis** — фоновая обработка статусов доставки/завершения заказов без блокировки HTTP-запросов.
- **Абстрактные классы платежей/доставки** (`BasePaymentGateway`, `BaseDeliveryGateway`) — лёгкое добавление CloudPayments/Boxberry без переписывания бизнес-логики.
- **Settings в БД, а не в `.env`** — администратор меняет комиссии, реферальные бонусы и пр. без передеплоя.
- **Zustand вместо Redux** — минимум бойлерплейта для авторизации и корзины.

## 🚀 Быстрый старт

### 1. Клонирование репозитория

```bash
git clone <your-repo-url> marketplace
cd marketplace
```

### 2. Настройка переменных окружения

```bash
cp .env.example .env
```

Откройте `.env` и заполните как минимум:

```ini
SECRET_KEY=сгенерируйте-длинную-случайную-строку-минимум-32-символа
YOOKASSA_SHOP_ID=ваш_shop_id_из_личного_кабинета_юкассы
YOOKASSA_SECRET_KEY=ваш_секретный_ключ_юкассы
CDEK_CLIENT_ID=ваш_client_id_сдэк
CDEK_CLIENT_SECRET=ваш_client_secret_сдэк
```

> **Важно:** Если ключи YooKassa/CDEK не заполнены — система продолжит работать:
> доставка рассчитывается по встроенной mock-формуле (200₽ + 100₽/кг), список
> ПВЗ вернёт несколько mock-точек, а платежи создаются с пустым
> `confirmation_url` (для тестирования бизнес-логики без реальных интеграций).
> Для продакшена ключи обязательны. То же касается `SMTP_HOST` — без него письма
> восстановления пароля не отправляются, а пишутся в лог backend-контейнера.

### 3. Запуск через Docker Compose

```bash
docker-compose up -d --build
```

Это поднимет стек микросервисов:
| Контейнер | Назначение | Порт |
|---|---|---|
| `marketplace_db` | PostgreSQL 17 | 5432 |
| `marketplace_redis` | Redis (брокер Celery) | 6379 |
| `marketplace_migrate` | One-shot: миграции + сидинг | — |
| `marketplace_gateway` | API-gateway (nginx, зеркало Kong) | 8000 |
| `marketplace_identity` | Сервис: auth, users, 2FA | (через gateway) |
| `marketplace_catalog` | Сервис: товары, магазины, отзывы | (через gateway) |
| `marketplace_orders` | Сервис: корзина, заказы, возвраты, доставка | (через gateway) |
| `marketplace_sellers` | Сервис: инструменты продавца, тарифы, промо | (через gateway) |
| `marketplace_platform` | Сервис: уведомления, лояльность, админка | (через gateway) |
| `marketplace_worker` / `_beat` | Celery worker / планировщик | — |
| `marketplace_frontend` | React (nginx) | 3000 |

Все API-вызовы идут на `http://localhost:8000/api/v1/*` → gateway разводит по сервисам.

### 4. Применение миграций

Миграции и сидинг выполняет **отдельный one-shot контейнер `migrate`** (общая БД),
все сервисы ждут его завершения. Команда — в `docker-compose.yml`:

```bash
/app/entrypoint.sh migrate && /app/entrypoint.sh seed
```

Если нужно прогнать вручную:

```bash
docker compose run --rm migrate sh -c "/app/entrypoint.sh migrate"
docker compose run --rm migrate sh -c "/app/entrypoint.sh seed"
```

### 5. Создание супер-администратора

Скрипт `seed.py` уже создаёт супер-админа автоматически из переменных `.env`:

```ini
FIRST_SUPERUSER_EMAIL=admin@marketplace.com
FIRST_SUPERUSER_PASSWORD=admin123
```

**Обязательно смените пароль после первого входа в продакшене.**

### 6. Доступ к приложению

| Что | Адрес |
|---|---|
| 🛍️ Витрина (фронтенд) | http://localhost:3000 |
| 🔧 Backend API + Swagger | http://localhost:8000/api/v1/docs |
| 👑 Админ-панель | http://localhost:3000/admin |
| ❤️ Health-check | http://localhost:8000/health |

Войдите в админку с `admin@marketplace.com` / `admin123`.

## 🧪 Тестирование основных сценариев

### Сценарий 1: Покупатель совершает покупку
1. Зайдите на http://localhost:3000/register, создайте аккаунт с ролью «Покупатель».
2. Перейдите в «Каталог» → добавьте товар в корзину (товары появятся после того, как
   продавец создаст их и они будут одобрены модератором, см. сценарий 2).
3. В корзине нажмите «Оформить заказ», укажите город и адрес.
4. Система рассчитает доставку через CDEK (или mock-формулу) и создаст платёж в YooKassa.
5. Без реальных ключей YooKassa вебхук не придёт автоматически — статус заказа можно
   изменить вручную через админку (`/admin/orders`) для тестирования жизненного цикла.

### Сценарий 2: Продавец публикует товар
1. Зарегистрируйтесь с ролью «Продавец» (или смените роль через `/admin/users`).
2. В `/seller` создайте магазин.
3. В `/seller/products` добавьте товар — он получит статус `pending` (если включена
   премодерация — `enable_premoderation=true` в настройках, по умолчанию включена).
4. Войдите как админ → `/admin/products` → отфильтруйте по «На модерации» → одобрите.
5. Товар появится в каталоге.

### Сценарий 3: Реферальная программа
1. В личном кабинете (`/referral`) скопируйте реферальную ссылку.
2. Зарегистрируйте второго пользователя по этой ссылке (`?ref=КОД` подставится
   автоматически в форму регистрации).
3. Новый пользователь совершает заказ на сумму ≥ `referral_buyer_min_order_amount`
   (по умолчанию 1000₽) и доводит его до статуса `completed`.
4. Пригласивший получает бонусные баллы — проверьте баланс в `/profile`.

### Сценарий 4: Управление комиссией
1. В `/admin/shops` найдите магазин и впишите индивидуальный процент комиссии прямо
   в таблице (inline-редактирование).
2. Либо измените глобальную комиссию в `/admin/settings` → вкладка «Комиссии и платформа».
3. Создайте новый заказ из этого магазина — `commission_percent_used` в заказе будет
   рассчитан по новым правилам автоматически.

### Сценарий 5: Жалобы и модерация
1. Авторизованный пользователь нажимает «Пожаловаться» на странице товара или
   магазина (`/shops/:id`).
2. Модератор видит жалобу в `/admin/reports`, рассматривает и проставляет резолюцию.

### Сценарий 6: Отзывы с модерацией, ответом продавца и лайками
1. Покупатель, чей заказ дошёл до статуса `completed`, оставляет отзыв на товар.
2. Если включена премодерация (`enable_review_premoderation`, по умолчанию `true`) —
   отзыв получает статус `pending` и не виден в каталоге.
3. Модератор одобряет отзыв в `/admin/reviews` — он появляется на странице товара,
   рейтинг товара пересчитывается.
4. Продавец отвечает на отзыв в `/seller/reviews` — ответ появляется под отзывом.
5. Другие покупатели отмечают отзыв как «полезно» — счётчик растёт, список
   отзывов сортируется по полезности.

### Сценарий 7: Восстановление пароля
1. На `/login` нажмите «Забыли пароль?» → введите email на `/forgot-password`.
2. Без настроенного `SMTP_HOST` ссылка для сброса не отправляется по почте, а
   пишется в лог: `docker compose logs identity | grep -i "reset"`.
3. Перейдите по ссылке вида `/reset-password?token=...`, задайте новый пароль.
4. Войдите с новым паролем.

## 🆕 Отзывы: модерация, ответы продавца, лайки

- **Модерация**: новые отзывы по умолчанию получают статус `pending` (управляется настройкой `enable_review_premoderation`, как и премодерация товаров) и не видны в каталоге, пока модератор не одобрит их в `/admin/reviews`. При отклонении обязательна причина. Рейтинг и счётчик отзывов товара пересчитываются только по `approved`-отзывам.
- **Ответы продавца**: продавец видит отзывы на свои товары в `/seller/reviews` и может оставить один ответ на каждый отзыв (с возможностью редактирования/удаления). Ответ виден покупателям рядом с отзывом на странице товара.
- **Лайки («полезно»)**: любой авторизованный пользователь может отметить одобренный отзыв как полезный; повторный клик снимает отметку (идемпотентно). Список отзывов на странице товара отсортирован по полезности.

## 🆕 Восстановление пароля

`POST /api/v1/auth/forgot-password` генерирует одноразовый токен с TTL 1 час и пытается отправить письмо со ссылкой `/reset-password?token=...`. Если SMTP не настроен (`SMTP_HOST` пуст в `.env`) — письмо не отправляется, а ссылка пишется в лог сервиса identity (`docker compose logs identity`), что удобно для тестирования без почтового сервера. Эндпоинт всегда возвращает успех независимо от того, существует ли email, чтобы не раскрывать базу зарегистрированных адресов.

## 🆕 Пункты выдачи СДЭК (ПВЗ)

При оформлении заказа покупатель может выбрать «Курьером» или «Самовывоз из пункта СДЭК» — во втором случае подгружается список ближайших ПВЗ через `GET /api/v1/delivery/pickup-points?city=...`. Без настроенных `CDEK_CLIENT_ID`/`CDEK_CLIENT_SECRET` возвращается несколько правдоподобных mock-точек, чтобы чекаут оставался тестируемым без реального доступа к API СДЭК.

## 🆕 Жалобы (форма в интерфейсе)

Кнопка «Пожаловаться» доступна на странице товара и на странице магазина (`/shops/:id`) для авторизованных пользователей. Жалоба создаётся через `POST /api/v1/reports` и попадает в очередь модератора (`/admin/reports`).

## 🆕 Исправление: заказы с товарами из нескольких магазинов

Раньше при оформлении заказа с товарами от разных продавцов комиссия считалась по ставке только одного (первого) магазина, а выплата при завершении заказа целиком уходила первому продавцу — второй продавец не получал ничего. Это исправлено:

- Комиссия теперь считается **по каждой позиции заказа отдельно**, с собственной ставкой её магазина (`OrderItem.commission_percent_used`, `OrderItem.platform_fee`, `OrderItem.seller_net`).
- При завершении заказа (`payout_service.payout_sellers_for_order`) **каждый продавец, представленный в заказе, получает свою долю** — выплата идёт по `shop_id`, а не по первой позиции. Эта же функция используется во всех трёх местах, где заказ может завершиться: покупателем, администратором вручную и автоматически по Celery-таймеру.
- `OrderItem.payout_status` (`pending` / `paid` / `refunded`) делает выплату идемпотентной: повторный вызов не задвоит начисление.
- Реферальное вознаграждение «за привлечённого продавца» также теперь срабатывает для **каждого** уникального магазина в заказе, а не только для первого.
- Исправлена дыра в авторизации: раньше любой продавец платформы мог сменить статус **чужого** заказа (проверялось только «является ли пользователь продавцом вообще», а не «есть ли его товар в этом заказе»).
- В личном кабинете продавца (`/seller/orders`) показывается **доля конкретного продавца** в заказе, а не полная сумма заказа; для заказов из нескольких магазинов есть пометка об этом. Для администратора (`/admin/orders`) детали заказа показывают разбивку выплат по каждому магазину отдельно.

**Известное ограничение, оставленное как есть:** статус заказа (`processing` → `shipped` → `delivered`) общий на весь заказ, а не на каждого продавца отдельно. Если в одном заказе товары от двух продавцов, первый продавец, отметивший свою часть как «отправлено», переводит в этот статус **весь** заказ — второй продавец технически может отправить свой товар позже, но интерфейс уже покажет заказ как отправленный. Корректное решение требует пере-проектирования заказа на под-заказы (`sub_order`) по каждому продавцу с собственным статусом и собственным трекинг-номером — это более крупное изменение модели данных, не входящее в это исправление. Финансовая часть (кому сколько и когда платить) корректна независимо от этого ограничения.

## 🆕 Службы доставки: СДЭК, Ozon, Яндекс, Почта России

Добавлены четыре службы доставки, каждая со своим gateway-классом (наследуют `BaseDeliveryGateway`) и реальной структурой API-запроса + mock-фоллбэком, если ключи не настроены. На чекауте покупатель видит **сравнение тарифов всех служб сразу** (`POST /api/v1/delivery/quote-all`) — цена и срок по каждой, отсортированные по стоимости — и выбирает удобную. У каждой службы свой набор ПВЗ. Ключи (`OZON_API_KEY`, `YANDEX_DELIVERY_TOKEN`, `RUSSIAN_POST_TOKEN` и др.) опциональны: без них работает mock-расчёт.

## 🆕 Платное размещение продавцов (тарифы)

Полноценная система тарифных планов:
- **Админ** в разделе «Тарифы» (`/admin/plans`) включает/выключает платное размещение одним переключателем (настройка `enable_paid_placement`) и управляет планами: название, **месячная цена**, **ставка комиссии этого плана**, **пробный период** (например, «первый месяц бесплатно» = `trial_days: 30`), план по умолчанию.
- **Продавец** в разделе «Тариф и комиссия» (`/seller/plan`) выбирает: бесплатный план с повышенной комиссией, либо платный со сниженной. Оплата подписки — **с баланса магазина** (из накопленной выручки) или **картой через ЮKassa** (на выбор).
- **Комиссия рассчитывается по приоритету**: индивидуальная ставка магазина (заданная админом) → ставка тарифа подписки → глобальная комиссия. Всё редактируется в админке.
- Сидируются три плана по умолчанию: «Бесплатный» (15%, 0 ₽), «Старт» (8%, 990 ₽/мес, 30 дней бесплатно), «Профи» (5%, 2990 ₽/мес, 14 дней бесплатно).

## 🆕 Настройки витрины магазина

Продавец оформляет публичную страницу магазина (`/shops/:id`): логотип, **баннер-шапка**, **акцентный цвет**, слоган, описание, контактные email/телефон. Всё это отображается на витрине, которую видят покупатели.

## 🆕 13 фич из анализа конкурентов

Большой блок возможностей, приближающий маркетплейс к Wildberries / Ozon / Etsy:

1. **Уведомления** — колокольчик в шапке с непрочитанными (статусы заказов, ответы на отзывы/вопросы, новые сообщения, выплаты, модерация). Эндпоинты `/notifications/*`.
2. **Промокоды продавца** — продавец создаёт свои купоны на свои товары в `/seller/coupons` (отдельно от глобальных админских). Таблица `seller_coupon`.
3. **Выводы средств** — продавец запрашивает вывод заработанного (`/seller/payouts`), админ одобряет/выплачивает (`/admin/payouts`); средства списываются с баланса только при статусе «выплачен». Таблица `payout_request`.
4. **Фото в отзывах** — к отзыву можно прикрепить изображения (таблица `review_photo`), показываются в карточке отзыва.
5. **Баннеры на главной** — управляются из админки (`/admin/banners`), показываются на главной. Таблица `homepage_banner`.
6. **Рекомендации «с этим покупают»** — на странице товара, на основе совместных покупок с фоллбэком на ту же категорию. Эндпоинт `/products/{id}/recommendations`.
7. **Варианты товара** — размер/цвет/SKU с собственной ценой и остатком (таблица `product_variant`). Корзина и заказы хранят `variant_id` и снимок `variant_name`. Товары без вариантов работают как прежде.
8. **Характеристики с фильтрацией** — админ заводит атрибуты (`/attributes`), продавец проставляет значения на товаре; используются в сравнении. Таблицы `attribute`, `product_attribute_value`.
9. **Чат покупатель ↔ продавец** — диалоги по магазину (`/chat`), один тред на пару покупатель-магазин. Таблицы `chat_thread`, `chat_message`.
10. **Вопросы к товару (Q&A)** — публичные вопросы на странице товара, продавец отвечает; обе стороны получают уведомления. Таблица `product_question`.
11. **Аналитика продавца** — графики выручки за 30 дней, топ-товары, итоги (`/seller/analytics`, recharts).
12. **Программа лояльности / кэшбэк** — за завершённый заказ покупателю начисляются баллы (% настраивается), идемпотентно. Расширяет бонусную систему. Настройки `enable_loyalty_cashback`, `loyalty_cashback_percent`.
13. **Сравнение товаров** — до 4 товаров, таблица характеристик (`/compare`), список в localStorage.

## 🚀 12 фич зрелости продукта

Блок, доводящий маркетплейс до уровня production-платформы:

1. **Возвраты и споры (RMA)** — покупатель оформляет возврат позиции заказа (`/returns`), продавец/админ одобряет/отклоняет/возвращает деньги (`/seller/returns`, `/admin/returns`). При статусе «refunded» средства идемпотентно возвращаются на баланс покупателя, а выплата продавцу реверсируется. Таблица `return_request`.
2. **Варианты в заказе** — корзина и оформление используют цену и остаток конкретного варианта; остаток списывается и возвращается с варианта, а не с базового товара.
3. **Под-заказы по продавцам** — мультимагазинный заказ разбивается на `sub_order` по магазинам, у каждого свой статус доставки и трек-номер; общий статус заказа выводится из наименее продвинутого под-заказа. `PATCH /sub-orders/{id}/status`.
4. **Автопродление подписок продавцов** — Celery beat ежедневно списывает оплату за тариф с баланса; при нехватке средств — даунгрейд на бесплатный план.
5. **Фасетные фильтры** — атрибуты с фильтрацией выводятся в каталоге (`/catalog/facets`), множественный выбор значений сужает выдачу.
6. **Email-уведомления** — важные in-app уведомления дублируются на email (best-effort, с учётом настройки `email_notifications` пользователя). В dev без SMTP — лог вместо отправки.
7. **Брошенная корзина + подписки на товар** — Celery-напоминание о забытой корзине; подписки «снова в наличии» и «снижение цены» (`/product-subscriptions`, `/my-subscriptions`) с рассылкой при срабатывании. Таблица `product_subscription`.
8. **2FA (TOTP)** — двухфакторная аутентификация на stdlib (без pyotp): секрет, QR-URL, резервные коды; вход требует код при включённой 2FA (`/2fa/*`, `/security`).
9. **Импорт товаров CSV** — массовая загрузка товаров продавцом (`/products/import-csv`, `/seller/import`) с построчным отчётом об ошибках.
10. **SEO** — sitemap.xml (`/sitemap.xml`), человекочитаемые slug у товаров (транслитерация кириллицы), генерация slug при создании.
11. **Мультивалютность** — курсы валют (RUB базовая, USD, EUR) в таблице `currency_rate`, переключатель в шапке, управление в админке (`/admin/currencies`). Цены хранятся в рублях, конвертация на отображении.
12. **Аналитика платформы** — дашборд админа (`/admin/platform-analytics`): GMV, доход платформы по комиссиям, динамика заказов/выручки за 30 дней, топ магазинов, рост пользователей.

Реализовано всё, разбито на блоки:

**Блок 5 — Админ:** когортная аналитика (retention heatmap, LTV, воронка), финансовая реконсиляция, гранулярный RBAC, аудит-лог с CSV-экспортом, feature flags с rollout %.

**Блок 6 — Инфраструктура:** стэкинг бонусов+промокодов с защитой от over-discount, загрузка файлов (локально или S3/MinIO через storage_service), шаблоны чата + часы работы магазина, юнит-тесты (pytest, чистая бизнес-логика), полнотекстовый поиск (MeiliSearch с ILIKE-фоллбэком).

### Известные ограничения после реализации
- **Web-push (пункт 5):** инфраструктура уведомлений готова (in-app + email), но браузерный web-push требует VAPID-ключей и service worker на фронте — оставлен как явный задел, не реализован полностью.
- **Полнотекстовый поиск:** работает на ILIKE из коробки; MeiliSearch подключается через настройки MEILI_URL/MEILI_KEY. Каталог использует свой ILIKE-фильтр; продвинутый поиск доступен через `/products/search`.
- **Загрузка файлов S3:** требует boto3 и настроек S3_* ; без них файлы сохраняются локально в UPLOAD_DIR.
- **Тесты:** покрывают чистую логику (комиссии, цены, slug, RBAC, приоритеты). Для интеграционных тестов с БД нужен тестовый MySQL/SQLite.
- Проект статически валидирован, но не запускался (нет сети/БД в среде сборки). Перед продакшеном: docker-compose up, alembic upgrade head, seed.py, функциональный прогон.
